## 1. CI/CD関連

### GitHub Actions とは？

**一言で**: GitHubに組み込まれた自動化ツール。コードをプッシュしたら自動でテスト・ビルド・デプロイできる。

**詳細説明**:
```
GitHub Actionsは、GitHubリポジトリで直接ワークフローを自動化できるCI/CDツールです。
YAMLファイルでワークフローを定義し、以下のようなトリガーで自動実行できます：
- コードのプッシュ
- プルリクエストの作成
- スケジュール実行（cron）
- 手動実行

例：プルリクエスト作成時に自動テスト→マージ時に自動デプロイ
```

**他ツールとの比較**:
| ツール | 特徴 |
|--------|------|
| GitHub Actions | GitHub統合、設定が簡単、無料枠あり |
| Jenkins | オンプレ向け、高カスタマイズ性、運用負荷高い |
| CircleCI | SaaS型、並列実行が得意 |
| GitLab CI | GitLab統合、オンプレ/SaaS両対応 |

**実務での使用例**:
```yaml
# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: npm run build
      - name: Deploy to ECS
        run: aws ecs update-service ...
```

---

### CI/CD とは？

**一言で**: コードの変更を自動でテスト・ビルド・デプロイする仕組み。

**詳細説明**:
```
CI (Continuous Integration) - 継続的インテグレーション
→ コード変更を頻繁にマージし、自動テストで品質を担保

CD (Continuous Delivery/Deployment) - 継続的デリバリー/デプロイ
→ テスト済みコードを自動で本番環境にデプロイ

メリット：
- 手動作業によるミスを削減
- リリースサイクルの短縮
- 問題の早期発見
```

**典型的なパイプライン**:
```
コードプッシュ → ビルド → 単体テスト → 統合テスト → ステージング → 本番
```

## 2. コンテナ・オーケストレーション関連

### Docker とは？

**一言で**: アプリケーションを「コンテナ」という箱に入れて、どこでも同じように動かす技術。

**詳細説明**:
```
Dockerは、アプリケーションとその依存関係を「コンテナイメージ」としてパッケージ化する技術です。

メリット：
- 環境差異の解消（開発環境と本番環境で同じ動作）
- 軽量（VMより起動が速い）
- ポータビリティ（どこでも動く）

主要概念：
- イメージ: アプリの設計図（読み取り専用）
- コンテナ: イメージから作られた実行環境
- Dockerfile: イメージを作るための手順書
```

**Dockerfileの例**:
```dockerfile
FROM node:18
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

---

### Kubernetes (K8s) とは？

**一言で**: 大量のコンテナを自動で管理・運用するシステム。

**詳細説明**:
```
Kubernetesは、Googleが開発したコンテナオーケストレーションツールです。
複数のコンテナを効率的に管理し、以下を自動化します：

- スケーリング: 負荷に応じてコンテナ数を自動調整
- セルフヒーリング: 障害が起きたコンテナを自動復旧
- ロードバランシング: トラフィックを複数コンテナに分散
- ローリングアップデート: 無停止でアプリを更新
```

**主要リソース**:
| リソース | 説明 |
|----------|------|
| Pod | コンテナの最小単位 |
| Deployment | Podのレプリカ数や更新を管理 |
| Service | Podへのアクセスを提供 |
| Ingress | 外部からのHTTPアクセスを制御 |
| ConfigMap | 設定情報を管理 |
| Secret | 機密情報を管理 |

---

### ECS vs EKS の違い

**一言で**: ECSはAWS独自のシンプルなコンテナサービス、EKSはKubernetesのマネージド版。

| 観点 | ECS | EKS |
|------|-----|-----|
| 学習コスト | 低い（AWS独自だがシンプル） | 高い（Kubernetesの知識必要） |
| ポータビリティ | AWSに依存 | 他クラウドへ移行しやすい |
| エコシステム | AWS中心 | 豊富なOSSツール |
| 運用負荷 | 低い | 中〜高い |
| ユースケース | シンプルなコンテナ運用 | 大規模・複雑な環境 |

**選定の判断基準**:
```
ECSを選ぶ場合:
- AWSに閉じた環境でOK
- シンプルに始めたい
- 運用リソースが限られる

EKSを選ぶ場合:
- マルチクラウド/ハイブリッドを視野に
- Kubernetesエコシステムを活用したい
- 既にKubernetesの知見がある
```

---

### Fargate vs EC2 の違い

**一言で**: Fargateはサーバー管理不要、EC2は自分でサーバーを管理。

| 観点 | Fargate | EC2 |
|------|---------|-----|
| サーバー管理 | 不要（サーバーレス） | 必要 |
| コスト | タスク単位課金（割高傾向） | インスタンス単位（スポット活用で安価） |
| スケーリング | 高速（秒単位） | やや遅い（分単位） |
| カスタマイズ | 制限あり | 自由度高い |
| GPU対応 | 限定的 | フル対応 |

## 3. AWS関連

### VPC とは？

**一言で**: AWS上に作る自分専用のプライベートネットワーク。

**詳細説明**:
```
VPC (Virtual Private Cloud) は、AWS上に論理的に分離されたネットワークを作成する機能です。

主要コンポーネント：
- サブネット: VPCを分割したネットワーク（パブリック/プライベート）
- インターネットゲートウェイ: インターネットとの接続口
- NATゲートウェイ: プライベートサブネットから外部へのアクセス
- ルートテーブル: トラフィックの経路を定義
- セキュリティグループ: インスタンスレベルのファイアウォール
- NACL: サブネットレベルのファイアウォール
```

**典型的な構成**:
```
VPC (10.0.0.0/16)
├── パブリックサブネット (10.0.1.0/24)
│   └── ALB, NAT Gateway
└── プライベートサブネット (10.0.2.0/24)
    └── EC2, RDS, ECS
```

---

### ALB vs NLB vs CLB の違い

**一言で**: ALBはHTTP/HTTPS向け、NLBは高性能TCP/UDP向け、CLBは旧世代。

| 観点 | ALB | NLB | CLB |
|------|-----|-----|-----|
| レイヤー | L7 (HTTP/HTTPS) | L4 (TCP/UDP) | L4/L7 |
| パス/ホストルーティング | ○ | × | × |
| WebSocket | ○ | ○ | × |
| 固定IP | × | ○ | × |
| 性能 | 中 | 高 | 低 |
| 推奨度 | ○ | ○ | 非推奨 |

---

### CloudFront とは？

**一言で**: AWSのCDN（コンテンツ配信ネットワーク）。世界中のエッジロケーションからコンテンツを高速配信。

**詳細説明**:
```
CloudFrontは、世界中のエッジロケーション（400+拠点）にコンテンツをキャッシュし、
ユーザーに最も近い場所から配信するCDNサービスです。

メリット：
- 低レイテンシ（ユーザー近くから配信）
- 高スループット（オリジン負荷軽減）
- DDoS対策（AWS Shield統合）
- HTTPS対応（ACM証明書統合）
```

---

### Lambda とは？

**一言で**: サーバーを用意せずにコードを実行できるサービス。使った分だけ課金。

**詳細説明**:
```
AWS Lambdaは、サーバーレスコンピューティングサービスです。
コードをアップロードするだけで、トリガーに応じて自動実行されます。

特徴：
- 自動スケーリング（同時実行数1000+）
- 従量課金（実行時間×メモリで課金）
- 多言語対応（Python, Node.js, Java, Go等）

ユースケース：
- APIバックエンド
- イベント駆動処理（S3アップロード時に画像加工等）
- 定期バッチ処理
```

## 4. 監視・SRE関連

### SRE とは？

**一言で**: Googleが提唱した、システムの信頼性を高めるためのエンジニアリング手法。

**詳細説明**:
```
SRE (Site Reliability Engineering) は、運用をソフトウェアエンジニアリングの
問題として捉え、自動化と効率化で信頼性を向上させるアプローチです。

主要概念：
- SLI (Service Level Indicator): 測定指標（可用性、レイテンシ等）
- SLO (Service Level Objective): 目標値（99.9%可用性等）
- SLA (Service Level Agreement): 顧客との契約
- エラーバジェット: 許容される障害の量

実践例：
- トイル（手作業）の削減・自動化
- ポストモーテム（障害振り返り）の実施
- カオスエンジニアリングの導入
```

---

### 可観測性（Observability）とは？

**一言で**: システムの内部状態を外部から理解できるようにする能力。

**3本柱**:
```
1. メトリクス (Metrics)
   - 数値データ（CPU使用率、リクエスト数等）
   - ツール: Prometheus, CloudWatch, Datadog

2. ログ (Logs)
   - イベントの記録（エラーログ、アクセスログ等）
   - ツール: ELK Stack, CloudWatch Logs, Splunk

3. トレース (Traces)
   - リクエストの経路追跡（分散トレーシング）
   - ツール: Jaeger, X-Ray, Zipkin
```

---

### Prometheus とは？

**一言で**: オープンソースの監視・アラートツール。Kubernetesとの相性が良い。

**詳細説明**:
```
Prometheusは、CNCFがホストする監視システムです。

特徴：
- プルモデル（Prometheusがターゲットからメトリクスを取得）
- 時系列データベース内蔵
- PromQL（強力なクエリ言語）
- Alertmanager統合

典型的な構成：
アプリ → Prometheus → Grafana（可視化）
           ↓
      Alertmanager → Slack/PagerDuty
```

---

### Grafana とは？

**一言で**: メトリクスを可視化するダッシュボードツール。

**詳細説明**:
```
Grafanaは、様々なデータソースに対応した可視化ツールです。

対応データソース：
- Prometheus, InfluxDB, Graphite
- CloudWatch, Datadog
- Elasticsearch, MySQL, PostgreSQL

主な機能：
- ダッシュボード作成
- アラート設定
- アノテーション（イベントマーク）
- 共有・エクスポート
```

## 5. IaC関連

### Terraform とは？

**一言で**: インフラをコードで定義・管理するツール。マルチクラウド対応。

**詳細説明**:
```
Terraformは、HashiCorpが開発したIaC（Infrastructure as Code）ツールです。

特徴：
- 宣言的記述（望む状態を記述）
- マルチクラウド対応（AWS, GCP, Azure等）
- 状態管理（tfstate）
- プラン→適用のワークフロー

基本コマンド：
- terraform init: 初期化
- terraform plan: 変更内容の確認
- terraform apply: 変更の適用
- terraform destroy: リソースの削除
```

**コード例**:
```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  tags = {
    Name = "web-server"
  }
}
```

---

### CloudFormation とは？

**一言で**: AWSネイティブのIaCサービス。YAMLまたはJSONでAWSリソースを定義。

**Terraformとの比較**:
| 観点 | Terraform | CloudFormation |
|------|-----------|----------------|
| 対応クラウド | マルチクラウド | AWSのみ |
| 言語 | HCL | YAML/JSON |
| 状態管理 | 外部（S3等） | AWS管理 |
| ドリフト検出 | 手動 | 自動 |
| 学習コスト | 中 | 低（AWS使用者） |

## 6. MLOps関連

### MLOps とは？

**一言で**: 機械学習モデルの開発から本番運用までを効率化する手法。DevOpsのML版。

**詳細説明**:
```
MLOpsは、機械学習システムの開発・デプロイ・運用を
継続的かつ効率的に行うためのプラクティスです。

主要コンポーネント：
- データパイプライン: データ収集・前処理の自動化
- 実験管理: モデルのバージョン・パラメータ管理
- モデルレジストリ: 学習済みモデルの保管・管理
- モデルサービング: 推論APIの提供
- 監視: モデル性能の継続的監視（ドリフト検出）
```

**典型的なパイプライン**:
```
データ収集 → 前処理 → 学習 → 評価 → デプロイ → 監視
    ↑                                          ↓
    └────────── フィードバック ←───────────────┘
```

---

### MLflow とは？

**一言で**: 機械学習の実験管理・モデル管理を行うオープンソースツール。

**主要機能**:
```
1. Tracking: 実験の記録（パラメータ、メトリクス、成果物）
2. Projects: 再現可能なMLコードのパッケージ化
3. Models: 様々な形式でのモデル保存・デプロイ
4. Registry: モデルのバージョン管理・ステージ管理
```

---

### SageMaker とは？

**一言で**: AWSのフルマネージドML開発プラットフォーム。

**主要機能**:
```
- SageMaker Studio: 統合開発環境
- SageMaker Training: 分散学習
- SageMaker Endpoint: モデルホスティング
- SageMaker Pipelines: MLパイプライン構築
- Feature Store: 特徴量の保存・共有
```

## 7. セキュリティ関連

### WAF とは？

**一言で**: Webアプリケーションを攻撃から守るファイアウォール。

**詳細説明**:
```
WAF (Web Application Firewall) は、HTTP/HTTPSトラフィックを検査し、
悪意のあるリクエストをブロックするセキュリティサービスです。

防御できる攻撃：
- SQLインジェクション
- クロスサイトスクリプティング (XSS)
- DDoS攻撃（L7）
- 不正なボットアクセス

AWS WAFの主要概念：
- Web ACL: ルールのコレクション
- ルール: 検査条件とアクション
- マネージドルール: AWSや3rdパーティ提供のルールセット
```

---

### IAM とは？

**一言で**: AWSリソースへのアクセスを制御するサービス。「誰が」「何を」できるかを管理。

**主要概念**:
```
- ユーザー: 人間のアカウント
- グループ: ユーザーの集合
- ロール: AWSサービスやアプリに付与する権限
- ポリシー: 権限を定義するJSONドキュメント

ベストプラクティス：
- 最小権限の原則
- MFA（多要素認証）の有効化
- ルートアカウントは使用しない
- アクセスキーは定期的にローテーション
```

---

### ゼロトラストとは？

**一言で**: 「何も信頼しない」を前提としたセキュリティモデル。常に検証する。

**詳細説明**:
```
従来の境界型セキュリティ: 社内ネットワーク内は信頼
ゼロトラスト: 場所に関係なく、常に認証・認可を要求

原則：
- 明示的な検証（すべてのアクセスを認証）
- 最小権限アクセス（必要最小限の権限のみ付与）
- 侵害の想定（侵害されることを前提に設計）
```

## 8. データベース関連

### RDS vs Aurora の違い

**一言で**: Auroraは高性能・高可用性に特化したRDSの上位版。

| 観点 | RDS | Aurora |
|------|-----|--------|
| 性能 | 標準 | MySQL比5倍、PostgreSQL比3倍 |
| 可用性 | マルチAZ（手動設定） | 自動フェイルオーバー |
| ストレージ | EBS | 独自（自動拡張、3AZ6コピー） |
| コスト | 安い | やや高い |
| リードレプリカ | 最大5台 | 最大15台 |

---

### インデックスとは？

**一言で**: データベースの検索を高速化する「索引」のようなもの。

**詳細説明**:
```
インデックスは、特定のカラムの値と行の位置を対応付けたデータ構造です。

メリット：
- SELECT（検索）が高速化

デメリット：
- INSERT/UPDATE/DELETEが遅くなる（インデックス更新が必要）
- ストレージ消費

作成すべきカラム：
- WHERE句で頻繁に使用されるカラム
- JOIN条件のカラム
- ORDER BY句のカラム
```
