# 技術用語解説 - 金融決済プラットフォーム基盤運用案件

本案件で頻出する技術用語の解説です。

---

## 1. SRE（Site Reliability Engineering）

### 概要
GoogleがDevOpsを実践するために開発したソフトウェアエンジニアリング手法。システムの信頼性向上を目的としたエンジニアリングプラクティス。

### 主な活動
- **SLI/SLO/SLA設計**: サービスレベル指標・目標・契約の定義
- **エラーバジェット管理**: 許容される障害レベルの管理
- **トイル削減**: 手動作業の自動化
- **インシデント対応**: 障害対応と事後分析（ポストモーテム）
- **キャパシティプランニング**: リソース計画

### 金融システムでの重要性
決済システムでは99.99%以上の可用性が求められることが多く、SREプラクティスによる信頼性向上が不可欠。

## 2. CI/CD

### CI（Continuous Integration）- 継続的インテグレーション
コードの変更を頻繁にメインブランチへ統合し、自動テストを実行するプラクティス。

### CD（Continuous Delivery/Deployment）- 継続的デリバリー/デプロイ
- **Continuous Delivery**: 本番環境へのデプロイをいつでも可能な状態に保つ
- **Continuous Deployment**: 自動的に本番環境へデプロイ

### 主要ツール
| ツール | 特徴 |
|-------|------|
| GitHub Actions | GitHubネイティブ、YAML定義、豊富なマーケットプレイス |
| Jenkins | オープンソース、プラグイン豊富、オンプレミス対応 |
| CircleCI | クラウドネイティブ、高速ビルド |
| GitLab CI | GitLabと統合、セルフホスト可能 |
| Cloud Build | GCPネイティブ |
| AWS CodePipeline | AWSネイティブ |

### パイプラインの構成例
```
コードプッシュ → ビルド → テスト → セキュリティスキャン → ステージング → 本番デプロイ
```

## 3. コンテナ技術

### Docker
アプリケーションとその依存関係をコンテナとしてパッケージ化する技術。

**主要概念**:
- **イメージ**: コンテナの設計図
- **コンテナ**: イメージから起動した実行インスタンス
- **Dockerfile**: イメージをビルドするための設定ファイル
- **レジストリ**: イメージを保存する場所（ECR、GCR、Docker Hub）

### Kubernetes（K8s）
コンテナオーケストレーションプラットフォーム。大規模なコンテナ環境の管理を自動化。

**主要概念**:
| リソース | 説明 |
|---------|------|
| Pod | 1つ以上のコンテナのグループ（デプロイの最小単位） |
| Deployment | Podのレプリカ管理、ローリングアップデート |
| Service | Podへのネットワークアクセスを提供 |
| Ingress | 外部からのHTTP/HTTPSトラフィック制御 |
| ConfigMap | 設定データの管理 |
| Secret | 機密情報の管理 |
| Namespace | リソースの論理的な分離 |

### マネージドKubernetes
| サービス | プロバイダー |
|---------|-------------|
| GKE (Google Kubernetes Engine) | GCP |
| EKS (Elastic Kubernetes Service) | AWS |
| AKS (Azure Kubernetes Service) | Azure |

## 5. 監視・オブザーバビリティ

### 三本柱
1. **メトリクス**: 数値データの時系列（CPU使用率、レスポンスタイム等）
2. **ログ**: イベントの記録
3. **トレース**: リクエストの経路追跡（分散トレーシング）

### 主要ツール
| ツール | 特徴 |
|-------|------|
| Prometheus | メトリクス収集、OSSの標準 |
| Grafana | ダッシュボード・可視化 |
| Datadog | 統合監視プラットフォーム |
| New Relic | APM、インフラ監視 |
| Cloud Monitoring | GCPネイティブ監視 |
| CloudWatch | AWSネイティブ監視 |

### アラート設計の原則
- **アクション可能**: 対応が必要なアラートのみ
- **ノイズ削減**: 閾値の適切な設定
- **エスカレーション**: 重要度に応じた通知先

## 6. IaC（Infrastructure as Code）

### 概要
インフラをコードで定義・管理する手法。バージョン管理、再現性、自動化を実現。

### 主要ツール
| ツール | 特徴 |
|-------|------|
| Terraform | マルチクラウド対応、宣言的記述 |
| CloudFormation | AWSネイティブ |
| Pulumi | 汎用プログラミング言語で記述 |
| CDK | プログラミング言語でCloudFormation生成 |

### Terraformの基本
```hcl
# プロバイダー設定
provider "google" {
  project = "my-project"
  region  = "asia-northeast1"
}

# リソース定義
resource "google_compute_instance" "default" {
  name         = "my-instance"
  machine_type = "e2-medium"
  zone         = "asia-northeast1-a"
}
```

## 7. 金融システム固有の用語

### PCI DSS（Payment Card Industry Data Security Standard）
クレジットカード情報を扱うシステムに求められるセキュリティ基準。

**主要要件**:
1. ファイアウォールの設置・維持
2. デフォルトパスワードの変更
3. カード会員データの保護
4. 暗号化による通信保護
5. マルウェア対策
6. システム・アプリの脆弱性管理

### FISC安全対策基準
金融情報システムセンター（FISC）が策定した、金融機関向けのセキュリティ基準。

### 決済システムの可用性要件
| レベル | 年間ダウンタイム |
|-------|----------------|
| 99.9% | 約8.76時間 |
| 99.99% | 約52.6分 |
| 99.999% | 約5.26分 |

## 8. GitOps

### 概要
Gitをインフラとアプリケーションのデプロイの信頼できる情報源（Single Source of Truth）として使用する運用モデル。

### 原則
1. **宣言的**: システムの望ましい状態をコードで記述
2. **バージョン管理**: すべての変更をGitで管理
3. **自動同期**: Gitの状態と実環境を自動的に同期
4. **承認プロセス**: Pull Requestによる変更レビュー

### 主要ツール
| ツール | 特徴 |
|-------|------|
| ArgoCD | Kubernetesネイティブ、UI付き |
| Flux | 軽量、CNCF incubating |

### 機能

- **セキュリティ**: mTLS（相互TLS）、認証・認可
- **オブザーバビリティ**: 分散トレーシング、メトリクス

### 主要ツール

## 10. バックオフィスシステム

### 概要
企業の内部業務を支援するシステム。顧客と直接接するフロントオフィスシステムの裏側で動作。

### 金融機関でのバックオフィス
- **決済処理**: トランザクション処理、清算
- **リスク管理**: 信用リスク、市場リスクの計算
- **コンプライアンス**: 規制報告、監査ログ
- **データ管理**: マスターデータ管理、レポーティング

### 特徴
- **高信頼性**: 24/365稼働、データ整合性
- **バッチ処理**: 日次・月次の大量データ処理
- **監査要件**: 詳細なログ、変更履歴の保持
